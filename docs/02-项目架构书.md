## 二、核心模块详细设计

### 2.1 智能交流中枢（Intelligent Communication Hub）

**定位**：整个软件最核心的交互板块，常驻于创作工作台中，是用户与AI进行朋友式聊天、构建世界观、管理所有创作要素的唯一入口。

#### 2.1.1 界面布局

```
┌─────────────────────────────────────────────────────┐
│ 智能交流中枢                          [身份选择 ▾]  │
├─────────────────────────────────────────────────────┤
│                                                     │
│  ┌─────────────────────────────────────────────┐   │
│  │                                             │   │
│  │           聊天消息区（主要区域）              │   │
│  │   - AI消息（含建议卡片、要素提取卡片）        │   │
│  │   - 用户消息                                 │   │
│  │   - 系统消息（要素更新通知、审核结果等）       │   │
│  │                                             │   │
│  │                                 [自动滚动]   │   │
│  └─────────────────────────────────────────────┘   │
│                                                     │
│  ┌─────────────────────────────────────────────┐   │
│  │ 输入区                           [发送] [🔧] │   │
│  └─────────────────────────────────────────────┘   │
│                                                     │
│  侧边标签：[要素提取历史] [待审核建议] [最近更新]     │
└─────────────────────────────────────────────────────┘
```

#### 2.1.2 AI角色身份系统

| 身份 | 语气风格 | 适用场景 |
|------|---------|---------|
| 知心小姐姐 | 温柔、鼓励、共情 | 灵感发散期、卡文期、需要情感支持时 |
| 知心小哥哥 | 幽默、轻松、伙伴感 | 日常写作陪伴、快速推进时 |
| 文学导师 | 专业、严谨、启发性 | 结构分析、技术性讨论、需要专业建议时 |

用户可随时切换身份，AI 根据身份调整对话风格，但底层能力不变。

#### 2.1.3 对话引导策略（心理学基础）

```
阶段1：共鸣建立（开场）
  "你这个设定好有意思！[具体指出一个亮点]，能多跟我说说[核心要素]吗？"
  心理原理：社会助长效应 + 叙事认同

阶段2：渐进深挖（每次聚焦1-2个维度）
  "你提到了[用户刚说的要素]，那[关联要素]是怎么运作的呢？"
  心理原理：渐进式披露 + 最近发展区理论

阶段3：主动建议（恰当时机插入）
  "我想到一种可能性，[具体建议]，你觉得呢？"
  心理原理：自主感维护（"你觉得呢"让用户有决策权）

阶段4：总结确认（阶段性收束）
  "我们刚才聊到了[要点总结]，我帮你整理好了，你看看对吗？"
  心理原理：胜任感增强 + 蔡格尼克效应（完成未完成事项的满足感）

阶段5：沉默尊重
  用户连续两次不接某话题 → 切换方向 → 从不穷追
  心理原理：自我决定理论（自主感保护）
```

#### 2.1.4 要素提取 Pipeline

```
用户消息/AI消息
      │
      ▼
┌──────────────┐
│ 意图识别      │ ← 判断是闲聊/设定补充/修改/提问/确认
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 实体抽取      │ ← 人物/地点/事件/物品/功法/势力/规则/关系/时间
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 关联匹配      │ ← 与现有Truth Files匹配：新增/更新/冲突/补充
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 提取卡片生成  │ ← 结构化展示：提取了哪些内容，分别属于哪个模块
└──────┬───────┘
       │
       ▼
  中枢聊天呈现 → 用户确认/修正 → 写入Truth Files → 级联更新关联模块
```

**关键设计原则**：提取结果以"卡片"形式展示在聊天中，明确标注"AI提取到以下内容"，用户可逐条确认/修改/拒绝，**不做静默写入**。

### 2.2 大纲系统

#### 2.2.1 大纲层级模型

```
作品 (Novel)
├── 卷 (Volume) — 可自定义命名（如"第一卷·初入江湖"）
│   ├── 章 (Chapter) — 每章有独立的章纲要
│   │   ├── 场景节拍 (Scene Beat) — 章节内的关键事件节点
│   │   └── 正文 (Content)
│   └── 章
└── 卷
```

#### 2.2.2 大纲初始化流程

```
用户创建新作品
      │
      ▼
填写初始大纲（自由文本，非表单）
  - 书名、类型
  - 一句话简介
  - 用户当前能想到的任何内容（不做完整性要求）
      │
      ▼
AI初步审核 → 评估大纲基础要素完整度
      │
      ├── 达到最低阈值 → "你的大纲基础框架已经有了，我们开始吧！"
      │         │
      │         ▼
      │   进入创作工作台（中枢开始引导完善）
      │
      └── 未达最低阈值 → "还差几个关键信息，我们聊聊？"
                │
                ▼
           中枢引导补充 → 补充完成 → 进入工作台
```

**"初始化"而非"终稿"**：用户的大纲永远是草稿态，AI帮助它动态生长，从不强迫用户一次性做完。

#### 2.2.3 章纲要系统（完整协作模式）

```
┌─────────────────────────────────────────────────────────┐
│              章纲要 — 双向自由入口                        │
│                                                         │
│  入口A：用户自己写纲要                                    │
│    ↓                                                    │
│    用户完成 → AI审核 → 中枢反馈建议                       │
│                                                         │
│  入口B：AI生成纲要草案                                    │
│    ↓                                                    │
│    AI基于"上级大纲 + truth files + 未回收伏笔"生成草案     │
│    → 推送到中枢 → 用户讨论调整                            │
│                                                         │
│  两条路径汇合                                            │
│    ↓                                                    │
│  用户确认 → 章纲要锁定为"当前执行版本"                     │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│              正文写作 — 三向自由入口                       │
│                                                         │
│  入口A：用户自己写                                        │
│  入口B：AI基于纲要+上下文生成正文草案                      │
│  入口C：混合模式（用户写一段→AI续→用户改→AI续）            │
│                                                         │
│  写作中：右侧面板常驻显示当前章纲要                        │
│  偏离>30% → 中枢主动提醒                                 │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│              章节完成 — 反向校准                          │
│                                                         │
│  Observer提取本章实际内容                                 │
│    ↓                                                    │
│  与章纲要对比分析                                         │
│    ↓                                                    │
│  生成"章节完成报告"                                      │
│    - 伏笔操作是否完成？                                   │
│    - 人物状态变化是否记录？                                │
│    - 主线推进程度？                                      │
│    - 实际vs纲要偏差分析                                  │
│    ↓                                                    │
│  偏差处理                                               │
│    小改 → 自动更新关联                                   │
│    中改 → 优化方案 → 中枢展示 → 用户确认                  │
│    大改 → 警告 + 详细影响分析 → 用户二次确认               │
└─────────────────────────────────────────────────────────┘
```

### 2.3 Truth Files 系统（结构化事实管理中心）

这是整个软件的数据基石。所有模块的数据最终都关联回 Truth Files，保证**单一事实源**。

#### 2.3.1 文件结构

```

├── truth_files/
│   ├── world_state.json        # 世界观当前状态

│   ├── resource_ledger.json    # 资源账本（功法/物品/货币/势力）
│   ├── pending_hooks.json      # 待回收伏笔清单
│   ├── chapter_summaries.json  # 每章摘要+关键事件
│   ├── subplot_board.json      # 支线面板
│   ├── emotional_arcs.json     # 情感弧线追踪
```

### 2.4 一致性校验引擎

#### 2.4.1 校验维度

借鉴 InkOS 的 33 维度审计，精简为适配网文场景的核心维度：

| 维度类别 | 具体检查项 | 严重程度 |
|----------|-----------|---------|
| **人物一致性** | 性格言行是否OOC | 致命 |
| | 能力/技能是否匹配当前状态 | 警告 |
| | 关系是否与人物矩阵一致 | 致命 |
| | 说话风格是否突变 | 建议 |
| **世界一致性** | 地点/环境描述是否与世界设定冲突 | 致命 |
| | 力量体系/规则是否被突破 | 致命 |
| | 势力关系是否前后矛盾 | 警告 |
| **剧情一致性** | 主线是否偏离大纲 | 警告 |
| | 伏笔是否有回收计划 | 建议 |
| | 因果链是否有断点 | 致命 |
| **结构质量** | 章节是否有有效冲突 | 建议 |
| | 是否有结尾钩子 | 建议 |
| | 节奏是否合理 | 建议 |
| **描写一致性** | 环境描写是否与人物心理协调 | 建议 |
| | 人物行动是否与心理动机一致 | 警告 |

#### 2.4.2 审核结果处理

```typescript
interface AuditResult {
  chapterId: string;
  issues: AuditIssue[];
  overallScore: number;          // 0-100
  isPassable: boolean;           // 是否可以发布/继续
}

interface AuditIssue {
  id: string;
  category: string;
  dimension: string;
  severity: 'critical' | 'warning' | 'suggestion';
  description: string;           // 问题描述
  location: string | null;       // 具体位置（段落引用）
  suggestion: string;            // 修复建议
  affectedModules: string[];     // 受影响的模块
}
```

#### 2.4.3 用户编辑保存时的审核流程

```
用户修改任何资料（大纲/人物/世界观等）
      │
      ▼
点击保存
      │
      ▼
变更内容推送到智能交流中枢
      │
      ▼
AI对比变更内容与所有现有资料
      │
      ├── 无冲突 → 直接保存 → 中枢通知"已保存，无冲突"
      │
      ├── 轻微冲突（影响<10%关联要素）
      │     → 保存 + 自动更新关联 + 中枢通知变更摘要
      │
      ├── 中度冲突（影响10-30%关联要素）
      │     → 生成优化方案 → 中枢展示 → 用户确认 → 批量更新保存
      │
      └── 严重冲突（影响>30%关联要素，可能颠覆体系）
            → 中枢严重警告 → 详细影响分析报告 → 用户二次确认
            → 确认 → 执行全面修复
            → 驳回 → 回滚本次修改
```

### 2.5 长文本处理策略

#### 2.5.1 三层上下文架构

```
┌─────────────────────────────────────────┐
│ Layer 1: 全局认知（~2K tokens）          │
│ - 全书摘要                               │
│ - 当前卷摘要                             │
│ - 核心人物当前状态一览                    │
│ - 待回收的重要伏笔（Top 10）              │
└─────────────────────────────────────────┘
              +
┌─────────────────────────────────────────┐
│ Layer 2: 结构化事实（~5K tokens）         │
│ - 当前章节相关的Truth Files切片           │
│ - 当前章纲要                              │
│ - 出场人物的完整当前状态                  │
│ - 相关伏笔的完整信息                      │
│ - 最近3章摘要                            │
└─────────────────────────────────────────┘
              +
┌─────────────────────────────────────────┐
│ Layer 3: 语义检索片段（~8K tokens）       │
│ - 基于当前写作内容语义检索最相关的         │
│   前文片段（通过向量相似度）              │
│ - 关键词精确匹配的特定要素               │
└─────────────────────────────────────────┘
              ≈
      总上下文：~15K tokens
  （远小于128K窗口，留有充足生成空间）
```

#### 2.5.2 RAG 混合检索策略

```
用户写作/中枢对话内容
        │
        ▼
┌──────────────────┐
│ 查询意图分析      │ ← 当前在写什么？涉及哪些人物/地点/事件？
└────────┬─────────┘
         │
    ┌────┴────┐
    ▼         ▼
┌────────┐ ┌──────────┐
│语义检索 │ │关键词检索 │
│(向量)  │ │(全文索引) │
└───┬────┘ └────┬─────┘
    │           │
    └─────┬─────┘
          ▼
┌──────────────────┐
│ 结果融合+重排序   │ ← 去重 + 相关度排序 + 多样性保证
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│ 上下文组装        │ ← 三层架构注入LLM
└──────────────────┘
```

#### 2.5.3 增量更新机制

```
每章完成后：
  Observer Agent 提取本章事实
      │
      ├── 更新 character_matrix（人物状态变化）
      ├── 更新 pending_hooks（伏笔操作）
      ├── 更新 chapter_summaries（新增本章摘要）
      ├── 增量更新所在卷摘要
      ├── 增量更新全书摘要
      ├── 更新 timeline
      ├── 更新 emotional_arcs
      └── 向量化本章内容 → 写入向量索引

全书摘要更新策略：
  - 不是每次都重新生成（成本高）
  - 使用增量摘要算法：
    旧全书摘要 + 新章摘要 → LLM合并 → 新全书摘要
```

