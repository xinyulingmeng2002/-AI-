## 一、项目概述

### 1.1 项目名称

**心御AI小说辅助器**（英文名：MindForge Novel Studio）

### 1.2 项目定位

一款以 **"智能交流中枢"为核心交互范式**、覆盖网文创作全流程的 AI 辅助编辑软件。区别于市面上"AI 代写"或"简单续写"类工具，本产品的核心差异化在于：

- **不是 AI 替你写，而是 AI 陪你写** —— 通过朋友式对话引导用户完成世界观构建、人物设定、伏笔追踪等认知负担重的环节
- **动态适配而非固定模板** —— 所有功能模块根据作品类型和用户需求动态生成与关联
- **五大创作阶段全覆盖** —— 从构思到输出的完整闭环，非单点功能拼凑

### 1.3 目标用户

- **第一版（MVP）**：个人使用，聚焦网文作者
- **第二版**：面向市场的商业化产品

### 1.4 核心痛点解决

| 痛点 | 现状 | 本产品方案 |
|------|------|-----------|
| 伏笔遗忘 | 作者自行记忆或手动笔记 | Truth Files 自动追踪埋设→呼应→回收全链路 |
| 人物OOC | 写着写着性格偏离设定 | 实时一致性校验，偏离时中枢主动提醒 |
| 主线偏离 | 剧情不知不觉跑偏 | 大纲锚定 + 偏离度检测 + 智能预警 |
| 设定枯燥 | 填表式世界观构建，逆反心理 | 朋友式聊天引导，利用沟通欲自然构建 |
| 环境/心理/行动脱节 | 描写缺乏内在一致性 | 场景-心理-行动三要素联动检查 |
| 长文本失控 | 几十万字后上下文丢失 | RAG + 分层摘要 + Truth Files 混合策略 |

## 三、技术路线

### 3.1 总体技术策略

```
共享核心代码（React + TypeScript）
    ├── 桌面端：Electron 封装
    └── 移动端：Capacitor 封装（Android + HarmonyOS）
```

### 3.2 技术栈选型

| 层级 | 技术 | 选型理由 |
|------|------|---------|
| 前端框架 | React 18+ / Vue 3 | 二选一，调研后决定。React生态更丰富（InkOS、AI-Novel-Writing均用React），Vue上手更简单（51mazi用Vue） |
| 桌面容器 | Electron | 跨平台Windows/macOS/Linux，生态成熟 |
| 移动端容器 | Capacitor | 与Web技术栈无缝衔接，支持Android+HarmonyOS |
| 编辑器 | TipTap (ProseMirror) | 51mazi验证过的成熟方案，支持富文本+Markdown混合 |
| AI编排 | LangChain / 自研轻量编排 | 模型路由、Prompt模板管理、Tool调用 |
| 向量数据库 | LanceDB / Qdrant | 轻量本地向量存储，支持RAG检索 |
| 本地数据库 | SQLite (better-sqlite3) | 轻量、可靠、无需服务端 |
| 状态管理 | Zustand / Pinia | 取决于React/Vue选型 |
| UI组件 | Radix UI / Element Plus | 取决于框架选型 |
| 样式 | Tailwind CSS | 快速构建，保持一致 |

### 3.3 模型路由层设计

```
模型路由层 (Model Router)
├── 通用适配器接口
│   ├── OpenAI-compatible adapter
│   ├── Anthropic adapter
│   ├── Google Gemini adapter
│   ├── DeepSeek adapter
│   ├── 国内模型适配器（智谱/通义/月之暗面等）
│   └── Ollama adapter（本地模型）
├── 任务级路由
│   ├── 中枢对话：创意型模型（高温）
│   ├── 纲要生成：平衡型模型（中温）
│   ├── 正文续写：创意型模型（中高温）
│   ├── 一致性审核：分析型模型（低温）
│   └── 要素提取：分析型模型（低温）
└── 用户配置层
    ├── API Key管理
    ├── 模型选择（按任务类型可选不同模型）
    └── 参数微调（温度、Top-P、最大长度等）
```

## 五、风险评估

| 风险 | 等级 | 应对措施 |
|------|------|---------|
| 长文本上下文管理效果不达预期 | 高 | 分层摘要+RAG混合策略，留有降级方案（简单截断+摘要） |
| 模型API成本过高 | 中 | 任务级路由分离高低成本模型，支持本地Ollama兜底 |
| 一致性审核误报率过高 | 中 | 审核结果分级（致命/警告/建议），用户可调节灵敏度 |
| 移动端性能问题 | 中 | MVP先不做移动端，降低早期复杂度 |
| 用户隐私顾虑 | 中 | 本地优先架构，API调用可选是否上传数据 |
| 单兵开发资源不足 | 高 | MVP严格控制范围，非核心功能坚决不做 |

## 六、成功指标

- **MVP核心指标**：用户能在30分钟内完成"创建作品→初始化大纲→中枢聊天完善设定→写一章→通过一致性审核"的完整流程
- **体验指标**：中枢聊天的要素提取准确率 > 80%（用户确认提取的内容无需修改即为准确）
- **质量指标**：一致性审核的致命问题召回率 > 90%
- **稳定性指标**：支持至少50万字的长篇作品，不出现性能退化或数据丢失
