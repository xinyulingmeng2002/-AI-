## 二、代码规范

### 2.1 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件名 | kebab-case | `chat-hub-panel.tsx` |
| 组件名 | PascalCase | `ChatHubPanel` |
| 函数/变量 | camelCase | `extractElements` |
| 常量 | UPPER_SNAKE_CASE | `MAX_CONTEXT_LENGTH` |
| 类型/接口 | PascalCase | `ChapterOutline` |
| 数据库表 | snake_case | `truth_files` |
| CSS类名 | kebab-case | `chat-message-card` |

### 2.2 TypeScript 规范

- 所有新代码必须使用 TypeScript
- 禁止使用 `any`，必须定义明确类型
- 接口以 `I` 开头不是必需的（使用 `interface` 关键字即可区分）
- 类型导入使用 `import type` 语法
- 所有公共函数必须有明确的返回类型注释

### 2.3 组件规范

- 一个文件只导出一个主组件
- 组件文件不超过 300 行（超过则拆分）
- Props 类型定义在组件文件顶部
- 优先使用函数组件 + Hooks，不使用类组件
- 避免深层嵌套（最多 4 层）

### 2.4 Git 提交规范

使用 Conventional Commits 规范：

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

| Type | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug修复 |
| `refactor` | 代码重构 |
| `style` | 样式修改 |
| `docs` | 文档更新 |
| `chore` | 构建/工具变更 |
| `perf` | 性能优化 |
| `test` | 测试相关 |

示例：
```
feat(chat-hub): 实现要素提取卡片展示

- 在聊天消息中插入提取卡片组件
- 支持逐条确认/修改/拒绝
- 确认后自动写入Truth Files
```

## 三、Git 工作流

### 3.1 分支策略

```
main ← 生产就绪代码
  └── develop ← 开发主线
       ├── feat/chat-hub    ← 功能分支
       ├── feat/truth-files
       ├── fix/audit-engine
       └── refactor/model-router
```

### 3.2 操作流程

```bash
# 1. 从 develop 创建功能分支
git checkout develop
git pull
git checkout -b feat/my-feature

# 2. 开发并提交
git add <specific-files>  # 不要用 git add .
git commit -m "feat(scope): description"

# 3. 推送前同步
git fetch origin
git rebase origin/develop  # 用 rebase 保持历史整洁

# 4. 推送
git push -u origin feat/my-feature

# 5. 创建 PR → 审核 → 合并到 develop
```

### 3.3 禁止操作

- **禁止** `git push --force` 到 main/develop
- **禁止** `git commit --amend` 已推送的提交
- **禁止** `git add .` 或 `git add -A`（必须手动确认添加到暂存区的文件）
- **禁止** 在 main 分支上直接提交

## 五、数据安全规范

### 5.1 API Key 管理

- API Key **绝对禁止**硬编码在代码中
- API Key 以加密形式存储在本地（使用 `safeStorage` API for Electron）
- API Key **不纳入**版本控制
- API Key **不纳入**云同步范围
- `.env` 文件加入 `.gitignore`

### 5.2 用户数据保护

- 所有用户作品数据默认本地存储
- 云端同步为用户主动开启的可选功能
- 传输中使用 HTTPS/TLS 加密
- 不同步：API Key、聊天历史、个人身份信息
- 向用户明确说明哪些数据会上传至AI服务商（写作内容摘要 vs 全文）

### 5.3 隐私设计原则

- 本地优先（Local-First）
- 最小化数据上传
- 用户可控（哪些数据上传、上传多少、传给哪个模型）
- 透明化（中枢对话中明确告知AI使用了哪些上下文数据）

## 六、测试规范

### 6.1 测试层级

| 层级 | 范围 | 工具 | 覆盖率目标 |
|------|------|------|-----------|
| 单元测试 | 核心逻辑函数 | Vitest | ≥ 80% |
| 集成测试 | API/模块间交互 | Vitest | ≥ 60% |
| E2E测试 | 关键用户流程 | Playwright | 覆盖核心路径 |

### 6.2 必须测试的模块

- **Truth Files 读写**：数据的完整性和一致性
- **要素提取 Pipeline**：各种输入格式的提取准确率
- **一致性校验引擎**：各类冲突的检出率和误报率
- **模型路由层**：各适配器的正确调用
- **章纲要审核**：大纲冲突检测

### 6.3 测试文件命名

```
src/truth-files/
├── world-state.ts
├── world-state.test.ts      # 与源文件同目录，.test.ts 后缀
└── __fixtures__/            # 测试数据
    └── sample-world.json
```

## 七、发布流程

### 7.1 版本号规范

遵循语义化版本（Semver）：`MAJOR.MINOR.PATCH`

- MAJOR：架构级变更，不兼容旧版本数据格式
- MINOR：新功能模块发布
- PATCH：Bug 修复和小优化

### 7.2 发布检查清单

- [ ] 所有测试通过
- [ ] TypeScript 编译无错误
- [ ] ESLint 无错误
- [ ] 在目标平台上手动测试核心流程
- [ ] 更新 `CHANGELOG.md`
- [ ] 更新版本号
- [ ] 打 Tag：`git tag -a v1.0.0 -m "Release v1.0.0"`
- [ ] 构建产物验证

### 7.3 Electron 构建

```bash
# Windows
pnpm electron:build --win

# macOS
pnpm electron:build --mac

# Linux
pnpm electron:build --linux
```

## 九、共享代码规范（v0.1 新增）

### 9.1 使用共享组件

以下组件已存在于 `components/shared/`，禁止在各页面重复实现：

- **Modal** — 所有模态对话框必须使用 `<Modal onClose={...}>...</Modal>`
- **EmptyState** — 所有空数据提示使用 `<EmptyState>内容</EmptyState>`
- **LoadingSpinner** / **LoadingText** — 统一加载状态展示

### 9.2 使用共享工具函数

`packages/shared/src/utils/` 提供：
- `generateId(prefix)` — 生成唯一ID（禁止各Store自行实现ID生成器）
- `clamp(value, min, max)` — 数值钳位
- `formatDate(dateStr)` — 统一日期格式 `zh-CN`
- `formatNumber(n)` — 带千分位数字格式

### 9.3 文档同步原则

**每次代码变更涉及架构/功能变化时，必须同步更新**：
1. `docs/01-项目计划书.md` — 功能状态、版本规划
2. `docs/02-项目架构书.md` — 新增模块、Store、组件
3. `docs/03-操作规范书.md` — 新规范、新共享资源
4. `progress/` — 阶段性进度报告

## 十、AI 交互规范

### 9.1 Prompt 管理

- 所有系统 Prompt 以独立文件存放（非硬编码）：`packages/core/src/prompts/`
- 每个 Prompt 有明确版本和用途注释
- 修改 Prompt 需记录变更原因
- 支持多语言 Prompt（中文优先）

### 9.2 模型调用日志（可选开启）

```
{应用数据目录}/logs/
├── model_calls/
│   ├── 2026-05-11.jsonl    # 每次模型调用的输入输出摘要
│   └── ...
```

- 日志仅本地存储
- 用户可随时清除
- 用于调试和优化 Prompt

### 9.3 Token 使用统计

- 仪表盘展示每月 Token 消耗
- 支持按任务类型/模型分别统计
- 可选设置月度预算上限
